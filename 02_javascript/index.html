Topic

Variables
In this section, we'll cover some of the particulars and peculiarities of variables in JavaScript.

Data Types

  • Primary (primitive) types — String, Number, Boolean
  • Composite types — Object, Array
  • Special types — Null, Undefined, NaN, Infinity
  • Weakly typed — variables can hold differing values over their lifetime
JavaScript defines three primitive, two composite, and a few special data types. We associate these types with data values, not variables, because JavaScript is weakly or loosely typed. That means a variable can hold varying values during its lifetime. Each primitive and composite data type is associated with an object that offers class methods. For example, String offers methods for extracting portions of strings. There are no object classes backing the special data types. JavaScript doesn't make distinctions between types of Numbers (i.e. integers, real, etc.), or Strings (char vs string).

Type Conversions

  • Automatic
  • Helper functions
  • Casting

var one = '1', two = 2, three='three';

// string concatenation has
// precedence over conversion
console.log(one + 1);  // = 11

// conversion when a NaN value might result
console.log(one * 2);  // = 2

// force a usage
console.log(parseInt(one) + 1); // = 2
console.log(String(1) + one); // = 11

// unless NaN would result
console.log(String(one) * 2); // = 2

isNaN(parseInt(three)); // == true
For a humorous look at what automatic type conversions can do, check out http://www.youtube.com/watch?v=_yZHbh396rc (JavaScript coverage begins about 1:20 but the whole video is just 4 mins long).

Variable Scope

  • Global scope
  • Function scope
  • Block scope (ECMA 6)
  • "Accidental" globals

var foo = 'bar'; // defined in the global scope
var fn = function() {
  var one = 1; // local to the function
  console.log(foo); // works! 
  oops = 'global'; // omitting 'var' defines
                   // this in the global scope
};
fn();  // run function, outputs 'bar'
console.log(oops); // outputs 'global'
console.log(one); // throws error
Default scope is global. Variables defined with 'var' in a function are local to that function; those without 'var' become globals. Functions have access to the scope of their parents, but parents don't have access to variables within sub-functions. ECMAScript 6 will support block scope with 'let' so you can define variables that are valid only within an if block, for example. Titanium doesn't support this yet.

Variable Hoisting

  • Var declarations moved to top
  • Var initializations are not
  • Function declarations are hoisted
  • Function statements are not

function baz(){}
var foo = 'bar';
baz();

// is treated like
function baz(){}
var foo;
baz();
foo = 'bar';

					

var foo = function() {
  bar();
};
foo(); // outputs 'I work!'
baz(); // 'baz is not a function'
function bar() {
  console.log('I work!');
}
var baz = function() {
  console.log('I do not');
};
Variable declarations are treated as if they were put at the top of the file, but not their initialiation (value assignment). The same is true for function declarations, in which case the entire assignment is moved to the top! That doesn't happen with function statements, though.

    console.log(arguments[2]);
  }

Pass by Reference or Value?

  • Simple variables passed by value
  • Objects passed by reference
  • Pedantically, everything is passed by value, but with objects that value is a reference to the object

function changeIt(x) {
  x = 5;
}
var x = 4;
console.log(x); // x is equal to 4
changeIt(x); 
console.log(x); // x is still equal to 4
					

var cat = {
  color: 'orange'
};
function changeCatColor(feline) {
  // obj reference passed in
  feline.color = 'black';
  feline.claws = 'sharp';
  feline = {
    color: 'brown'
  };
}
changeCatColor(cat);
console.log(cat.color); // outputs 'black'
console.log(cat.claws); // outputs 'sharp'
You have to be aware of what goes on when passing values to functions. In practice, it's easiest to think about it as: simple variables are passed by value and objects are passed by reference. Technically, everything is passed as a value to the function. In the case of passing objects, that value happens to be a reference to the object. What this means is that you can change the contents of the passed-in object, even add members. But you can't replace the object itself.

What is "this"?

  • Refers to the object associated with the function
  • When functions are methods, 'this' = the object
  • With 'normal' functions, 'this' = the object, but the object is the global object

var obj = {};
obj.fn = function () { 
  // 'this' == obj
}; 
function fn() { 
  // this == global object
};
// because we're basically saying
window.fn = function();
The value of 'this' in functions is often perceived to be confusing. But really it's fairly straightforward once you understand that 'this' was added so that constructors could refer to the object with which they are associated. In other words 'this' was intended to be used with functions that are properties of objects. The confusion happens because people forget that all functions (or variables) are associated with an object -- if not explicitly specified then they're associated with the global object. In a browser, that's the 'window' object; in Titanium, it's a generic object created to act as the global.

Using 'call'

  • Defines what 'this' is for the function
  • Without call, you're essentially passing the global object with call

function hello(thing) {
  console.log(this + " says hello " + thing);
}
hello.call("Bart", "world") // => Bart says
                              // hello world

// without call
hello('world');

// essentially means
hello.call(window, 'world');
call lets you specifically name the object that will become 'this' within a function. It's useful when you create objects in the global scope, but want to control 'this' during execution.

Self-calling Functions

  • Define, then execute, a function
  • Wrap function definition in ()
  • Add () at the end (to call it)
  • Works with function statements

// define & declare Counter
var Counter = (function() {
  var privateCounter = 0;
  return {
    increment: function() {
      privateCounter++;
    },
    decrement: function() {
      privateCounter--;
    },
    value: function() {
      return privateCounter;
    }
  }; 
})();
Self-calling functions are sometimes called immediately executing functions. You define them, and call them in a single statement. They are a convenience only; you save a line or two of code declaring then calling the function. They are useful with object constructors as shown.

Closures

  • Variables within functions go out of scope when the function ends
  • Closures let you preserve a local variable's value
  • In this example, increment(), decrement(), and value() are all closures
  • Closures can cause "memory leaks"

var Counter = (function() {
  var privateCounter = 0;
  return {
    increment: function() {
      privateCounter++;
    },
    decrement: function() {
      privateCounter--;
    },
    value: function() {
      return privateCounter;
    }
  }   
})();
// Counter fn is done, object returned
// but 'privateCounter' still valid
Counter.increment();
console.log(Counter.value()); // outputs 1
Counter.decrement();
console.log(Counter.value()); // outputs 0
Variables are local to functions, and go out of scope when the function ends. But if the function remains valid, its variables remain valid. So in the example, the outer function ends, thus you might think privateCounter would go out of scope. Yet the returned object's methods retain references to that variable, so it must remain valid.

// var person = new Object();

Functions as Constructors

  • Functions that create objects
  • Use care omitting 'new'
  • 'this' == object being instantiated with 'new'
  • 'this' == global object without 'new'

function Person(name) {
  this.name = name;
}

var jim = new Person('Jimmy');
console.log(jim.name); // outputs 'Jimmy'

var jane = Person('Jane');
console.log(jane.name); // throws error
// in browser environment:
console.log(window.name); // outputs 'Jane'
Remember, there are no classes. So you don't define then instantiate a class. You call a function to create an object. Any function that returns an object is a constructor as far as JavaScript is concerned.

Implicit vs. Explicit Returns

  • Implicit = no return statement
  • Used with a constructor, returns the object that was created
  • Implicit returns can cause problems — if you later forget 'new'

function Person(name) {
  this.name = name;
}
function Android(name) {
  return {
    name: name
  };
}

var jim = new Person('Jimmy');
alert(jim.name); // => Jimmy
var jane = Person('Jane');
alert(jane.name); // => undefined!
alert(window.name); // => 'Jane'

var data = new Android('Cmdr Data');
console.log(data.name);  // => Cmdr Data
var lore = Android('Lore');
console.log(lore.name); // => Lore
An object is implicitly returned if you don't include an return statement in your function. Thus, Person() and Android() both return an object with a name property.

However, problems arise with implicit returns if you forget the new keyword when instantiate an object. If you omit new, you still get an object but its properties will be associated with the global object (shown here as window). As shown, explicit returns eliminate this problem.

Prototypal Inheritance

"I have learned to fully embrace prototypalism, and have liberated myself from the confines of the classical model." -- Douglas Crockford

  • JavaScript is a class-free, object oriented language
  • Inheritance is via prototypes, not classes
  • A prototype is an object from which other objects inherit properties
  • Any object can be a prototype
  • Every object has a prototype except the top-level Object
  • __proto__ is a way to access an object's prototype
Rather than trying to force-fit classical inheritance on JavaScript, you should use the language as it was designed to be use. JavaScript can provide for inheritance, class & instance methods, and so forth. It does so via prototypes.

Objects inherit properties and methods from their prototype. Class properties and methods can be added to the prototype; instance methods to the object itself. You don't define a class then extend that class to instantiate. We'll see examples on the upcoming slides. We'll also talk about how to use that proto property.

Prototype Example


function Parent() {
  this.whoami = 'parent';
}
Parent.prototype = {
    speak: function() {
      console.log('Hello');
    }
};
// Child will inherit from Parent
function Child() {}

// Set Child's prototype to a new instance of Parent
Child.prototype = new Parent();
Child.prototype.bye = 'Goodbye';

var son = new Child(); // create an instance

console.log(son instanceof Child); // true
console.log(son instanceof Parent); // true
son.speak(); // says 'Hello'
console.log(son.whoami); // 'parent'
Here we define a constructor, and modify it via its prototype property. Next, we define an object that will inherit from Parent. We create the association by pointing the child's prototype to be an instance of the parent. We add a property to the parent (ie. the class), then check out the result with some logging statements. Ref: http://javascriptweblog.wordpress.com/2010/06/07/understanding-javascript-prototypes/

Inheritance - Property/Method Lookup

  1. Looks first at specific object
    son.hasOwnProperty(whoami)==false
  2. Looks to up the prototype chain
    Child.hasOwnProperty(whoami)==false
    Parent.hasOwnProperty(whoami)==true
  3. Continues till it reaches Object, the prototype of all objects. If not there, returns undefined

function Parent() {
  this.whoami = 'parent';
}
Parent.prototype = {
    speak: function() {
      console.log('Hello');
    }
};
function Child() {}
Child.prototype = new Parent();

var son = new Child();
console.log(son.whoami);
console.log(son.foo); // undefined
So how does JavaScript find properties and methods you call on? It looks up the prototype chain. Every object has a prototype except the top-level Object itself. In this way, children can override or hide properties and methods of the parent. Ref: https://developer.mozilla.org/en-US/docs/JavaScript/Guide/Inheritance_and_the_prototype_chain

Changing Prototypes


function Person() {
  this.whoami = 'human';
}
function Alien() {
  this.whoami = 'martian';
}
var fred = new Person();
// force Fred's prototype to be Alien's prototype
fred.__proto__ = Alien.prototype;

console.log(fred instanceof Alien); // true

// we can add new properties
Alien.prototype.color = 'green';
alert(fred.color); // green

// but original values are preserved
alert(fred.whoami); // human!

// the Person connection is now broken
Person.prototype.arms = 2;
alert(fred.arms); // undefined
We can manually break parent/child associations by changing an object's prototype. In this situation, we have to set the child object's __proto__ property to be the prototype property of the parent. See the next slide for more info on the differences between proto and prototype.

__proto__ vs obj.prototype

  • Every function has a prototype property
  • It refers to the object that will be assigned as the prototype of the resulting object
  • Thus obj.__proto__ refers to an object's parent
  • and obj.prototype refers to the object that is created by the parent constructor

var Circle = function(radius) {
    this.radius = radius;
    //next line is implicit
    //this.__proto__ = Circle.prototype;
}

// augment Circle's default prototype property
// thereby augmenting each new instance
Circle.prototype.area = function() {
   return Math.PI*this.radius*this.radius;
}

var a = new Circle(3);
alert(a.area().toFixed(2)); //28.27

alert(a.__proto__ == a.prototype); // false

alert(a.__proto__ == Circle.prototype); // true
While the difference is subtle, it is important. Functions have a prototype property; objects have a __proto__ property. A function's prototype property points to the object (ie. the class) that will be returned. An object's proto property points to its parent. To add new class methods, either assign them to the function's prototype property or an instance's __proto__ property.

Ref: http://javascriptweblog.wordpress.com/2010/06/07/understanding-javascript-prototypes/

Parasitic Inheritance

Simple pattern for inheritance in JavaScript.


var Person = function(name, age) {
  this.name = name;
  this.age = age;
};
var Employee = function(name, age, group) {
  var e = new Person(name, age);
  e.group = group;
  return e;
};
var alex = new Employee('Alejandro', 25, 'Developer');

(Works with __proto__ so you don't have to!)

This syntax is perhaps the more typical way that developers will subclass or extend objects. Here we see two constructors. The second instantiates a Person object, adds properties, then returns it. This is simpler code than working directly with __proto__ and prototype, and thus eliminates a bunch of confusion. Ref: http://daniel.carrera.name/2008/02/parasitic-inheritance-in-javascript/

Modules

CommonJS

  • Standard way to create self-contained, exportable functionality. See commonjs.org for the spec.
  • Goals:
    • Private scope within the module
    • Explicitly exported API
    • Decoupling and reusability
  • Inside a module, exports is an object to which you can attach properties
  • You can replace that property by setting module.exports
  • Modules "imported" with the require() global function
Modules are meant to solve JavaScript's limited notion of scope. Each module has its own scope (or is supposed to) and must explicitly export variables to make them available to the calling context.

Helper / Library Pattern


// file is network.js

exports.getData = function() {
  // get data from the network
};
exports.postData = function() {
  // post data to the network
};




// how to use it

var net = require('network'); // no '.js'!!

var someData = net.getData();
button.addEventListener('click', function() {
  net.postData(someData);
});
This pattern is useful for libraries of helper functions, such as network or database code.

Factory Pattern


// file is Button.js

// creates a UI component we'll need many copies of
var Button = function(title) {
	return Ti.UI.createButton({ title: title });
};

module.exports = Button; // return the constructor




// how to use it

var Button = require('ui/Button');

// now make some instances
var button1 = new Button('foo');
var button2 = new Button('bar');
By assigning a function to the module.exports variable, you can use the returned object as a constructor. Great for UI components.

We're not going to dig into the many other patterns. You can direct curious students to the 'Design Patterns in JavaScript' site for further information and examples: https://github.com/tcorral/Design-Patterns-in-Javascript (that site itself is based on the book described at http://en.wikipedia.org/wiki/Design_Patterns).

Modules in Titanium

  • Caching — module isn't re-evaluated with subsequent require()'s
  • Omit leading '/' ... require('ui/somemodule')
  • Paths are relative to Resources directory
  • Pass in live data the module will need (e.g. a user object)
  • Require in any libraries/modules the module will need (e.g. your database module)
  • Platform gotcha -- on iOS, a module has access to its parent's global object -- but don't count on it
In Titanium, modules are cached. So re-requiring a module (such as you database library) won't incur any sort of memory or performance hit. Paths are entered without the leading slash, relative to the Resources directory. As of this writing, modules on iOS have access to the caller's global object. That might change. You don't need it anyway--just pass in or require in the objects you need.

Garbage Collection

  • Automatic via "mark & sweep"
  • Objects collected when no references remain
  • Nothing in the global space is ever collected!
  • "Force" collection by encapsulating within functions or modules that will go out of scope
Before we can talk about avoiding memory leaks, we need to understand how JavaScript cleans up variable references and memory.

JavaScript uses a "stop the world" automatic garbage collection routine to clean up memory. It sweeps through memory, marking objects in use; then tears down any that are unmarked (and hence unused). Nothing in the global scope can be collected because there will always be a reference to the object. Within functions or other scopes, you can force collection by encapsulating within functions or modules that will go out of scope.

Avoiding Memory Leaks

  • Don't store variables in the global scope
  • Set objects = null when no longer needed
  • Watch for "hidden" references in functions, closures, event listeners, accidental global assignments
  • Profile your app
These are some techniques to minimize wasted memory and avoid leaks. Setting objects to null doesn't actually remove the object. But it minimizes it's memory use. We'll cover how to profile an app in the advanced course.

Performant Coding

  • Defer execution — load modules as needed & don't Ti.include() code
  • Modules cache code so it's not re-evaluated
  • Avoid the global scope
  • Use closures sparingly
  • Avoid eval() and implicit evals
  • Don't change data types stored in a variable
Avoid include() statments and use modules instead. They offer deferred execution and caching to both minimize memory use and speed up perceived performance. Avoid the global scope. In addition to memory leak problems, JavaScript must "step up" through the scope chain to find your variables. This means you should define variables a function needs inside that function when feasible.

Modern JavaScript engines like V8 include internal optimizations that can be defeated when you change the type of data stored in a variable. So while JavaScript permits that, try to not do it.

Performant Loops

  • Set upper bound before loop
  • For/in loops slower than for or while
  • Define local references
  • Operators a little faster than functions

for(var i=0, j=arr.length; i < j; i++) { }
// faster than
for(var i=0; arr.length; i++) { }

var foo = bar.someproperty.anotherproperty
while(foo < 10) { }
// faster than
while(bar.someproperty.anotherproperty < 10) { }

for(var i=0; i < 100; i++) {
  someVal = (a < b) ? a : b;
  // is a tiny bit faster than
  someVal = Math.min(a,b);

  arr[arr.length] = newMember
  // is faster than
  arr.push(newMember)
}
Loops by their nature are very sensitive to even small optimizations. Thus, take the time to implement features like shown here. The last bullet shown in code: while operators are only a tiny bit faster than functions, in a long-running loop this difference matters.

Event listeners are slower than callbacks — I realize we haven't dug into event listeners yet. But keep in mind that you should use callbacks instead.

Titanium and Alloy offer build-time optimization techniques that create much faster code than testing to see if you're running on Android or iOS. Use those techniques.

Don't extend the built-in Titanium objects. You app could crash or hang.
